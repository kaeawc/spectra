# Snapshots and hosts

Spectra's snapshot registry connects four concepts that otherwise show
up in separate design docs:

- **Host identity**: one Mac, keyed by machine UUID.
- **Snapshots**: timestamped captures of that host's state.
- **Baselines**: named, immutable snapshots used as reference points.
- **Diffs**: structural comparisons between two snapshots, often across
  hosts.

The implementation lives in the local SQLite database at
`~/.spectra/spectra.db`. See [Storage](../design/storage.md) for the
table layout and [System inventory](../design/system-inventory.md) for
the full JSON shape stored with each snapshot.

## Host identity

Spectra uses macOS's `IOPlatformUUID` as the primary host identity. The
collector reads it from:

```bash
ioreg -d2 -c IOPlatformExpertDevice
```

That value becomes `snapshot.host.machine_uuid` and the primary key in
the `hosts` table. Hostname is recorded too, but it is descriptive: it
can change when the Mac is renamed, joins a different network, or appears
through Tailscale DNS.

If the machine UUID cannot be collected, Spectra falls back to hostname
when persisting the row. That keeps snapshots usable on partial or
restricted systems, but those rows are weaker identities and should not
be treated as a stable physical-machine key.

## Registry lifecycle

Every stored snapshot upserts the host row and inserts one snapshot row:

1. `spectra snapshot` builds a live inventory.
2. `hosts.machine_uuid` is inserted or updated with the latest hostname,
   OS, CPU, memory, architecture, and `last_seen`.
3. `snapshots.id` is inserted with `machine_uuid`, `taken_at`, `kind`,
   optional `name`, Spectra version, app count, and the full
   `snapshot_json`.
4. Query-focused child rows are written for apps, processes, login
   items, and granted permissions.

The host row is the registry entry. It is not manually created; it
appears after the first stored snapshot from that machine.

```bash
spectra snapshot
spectra hosts
spectra snapshot list
```

`spectra hosts` lists the known local registry. `--discover` can merge
Tailscale peer names from `tailscale status --json`, and
`--discover-daemons` keeps only peers with a reachable Spectra daemon.
Discovered peers are transient command output unless a snapshot from
that host is stored locally.

## Snapshot IDs

Snapshot IDs are generated by the snapshot package in this form:

```text
snap-YYYYMMDDTHHMMSSZ-random
```

The timestamp makes IDs sortable and readable. The random suffix avoids
collisions when multiple snapshots are taken in the same second.

Use `spectra snapshot list` to find IDs and
`spectra snapshot show <id>` to inspect one stored snapshot.

## Live snapshots

Live snapshots are ordinary stored captures with `kind = live`.

```bash
spectra snapshot
spectra snapshot create
spectra snapshot --no-apps
spectra snapshot --network
```

Use `--no-store` when a snapshot should be printed but not registered:

```bash
spectra snapshot --no-store --json
```

The `live` sentinel in diff commands is different from a stored live
snapshot. It captures the current machine state in memory and does not
write a new registry row:

```bash
spectra diff snap-20260504T095749Z-4829 live
```

## Baselines

A baseline is a stored snapshot with `kind = baseline` and an optional
human name. Baselines are immutable reference points and are skipped by
live-snapshot pruning.

```bash
spectra snapshot create --baseline pre-incident
spectra snapshot create --baseline --name pre-upgrade
spectra baseline list
spectra baseline drop snap-20260504T095749Z-4829
```

Baseline names are labels, not primary keys. Diff resolution checks a
baseline ID first, then a baseline name:

```bash
spectra diff baseline pre-incident live
```

When the name or ID is omitted, Spectra uses the newest baseline:

```bash
spectra diff baseline
```

## Diff operands

`spectra diff` compares two snapshots. Operands can be:

| Operand | Meaning |
|---|---|
| `snap-...` | Stored snapshot ID in the local database |
| `live` | Fresh, unstored local snapshot |
| `baseline [name|id] [other]` | Baseline-aware shorthand |
| `<host>` | Newest stored snapshot for a known host |
| `<host>@<snapshot-id>` | Explicit snapshot from a host |

Examples:

```bash
spectra diff snap-20260504T095749Z-4829 snap-20260505T100012Z-7341
spectra diff workstation work-mac
spectra diff workstation@snap-20260504T095749Z-4829 work-mac@snap-20260503T170201Z-9932
spectra diff baseline pre-incident live
```

Diff comparison is structural, not line-oriented. Apps are matched by
bundle ID, JDKs by version/vendor identity, Homebrew formulae by name,
listening ports by `(port, proto)`, and `$PATH` by ordered sequence. See
[Diff semantics](../design/system-inventory.md#diff-semantics) for the
category rules.

## Retention

Live snapshots are pruned by count. The default retention keeps the
newest 100 live snapshots per machine. Baselines are never pruned by this
path.

```bash
spectra snapshot prune
spectra snapshot prune --keep 25
```

Use a baseline for any state that must survive routine cleanup:

```bash
spectra snapshot create --baseline before-jdk-upgrade
```

## Remote hosts

The local database remains the source of truth for registered hosts.
Remote daemon calls can create, list, fetch, or diff snapshots on another
Spectra daemon, but each daemon owns its own SQLite database.

For engineer-to-engineer debugging:

1. Each Mac runs `spectra snapshot` or exposes `snapshot.create` through
   the daemon.
2. `spectra hosts` shows machines already seen in the local registry.
3. `spectra hosts --discover-daemons` finds reachable Tailscale peers
   running Spectra.
4. `spectra diff <host-a> <host-b>` compares the newest stored snapshot
   for each host when those snapshots are reachable or already known.

The storage model deliberately avoids multi-writer replication. Cross-host
diff is a client operation that loads two snapshots and compares them.
