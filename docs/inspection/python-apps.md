# Python-based app inspection

Python macOS apps usually present as native `.app` bundles, but the real
runtime can be hidden behind launchers generated by py2app, PyInstaller,
Briefcase, Nuitka, or a custom shell/native stub. Spectra treats Python app
inspection as enrichment on top of the normal framework verdict: a bundle can
still classify as AppKit, Qt, or Unknown while also exposing a Python runtime,
packaging tool, and dependency footprint.

The goal is to answer:

- Which Python runtime will this app actually start?
- Is the runtime bundled, system-provided, Homebrew-provided, or manager-owned?
- Which packaging tool shaped the bundle?
- What dependency and native-extension surface is embedded?
- Does the app rely on writable code paths, user-site packages, or ambiguous
  environment resolution?

## Bundle signals

Static inspection starts with low-cost bundle markers under `Contents/`:

| Signal | Interpretation |
|---|---|
| `Resources/__boot__.py`, `Resources/site.py`, `Resources/lib/python*/` | py2app-style bundle |
| `MacOS/<name>` plus `Resources/base_library.zip` and `Resources/lib-dynload/` | PyInstaller-style bundle |
| `Resources/app_packages/`, `Resources/app/`, `Resources/python/` | Briefcase-style bundle |
| `Frameworks/Python.framework/` | bundled CPython framework |
| `Frameworks/libpython*.dylib` | embedded or sidecar CPython runtime |
| `*.pyc`, `*.pyo`, `__pycache__/`, `*.dist-info/`, `*.egg-info/` | Python bytecode or package metadata |
| `.so` files under Python package paths | native extension modules |

No single generic Python marker should override a stronger UI-framework
classification. For example, a Qt app with Python packages remains `UI=Qt`
and gains Python inspection details; an AppKit launcher with PyInstaller
markers can be reported as Python-backed even when Layer 2 only sees AppKit.

## Runtime resolution

Spectra should record the interpreter source separately from the app verdict:

```go
python_app {
  packaging       # py2app | pyinstaller | briefcase | nuitka | custom | unknown
  runtime_source  # bundled-framework | bundled-dylib | bundled-executable | system | brew | pyenv | uv | mise | asdf | unknown
  runtime_path    # bundle-relative when inside the app, absolute otherwise
  version         # parsed from binary metadata, pyvenv.cfg, or package layout
  module_roots[]  # bundle-relative import roots
}
```

Resolution order should be deterministic:

1. Prefer an interpreter or CPython library inside the `.app` bundle.
2. Inspect launch scripts and launcher metadata for explicit interpreter paths.
3. Fall back to `pyvenv.cfg`, `site.py`, `orig-prefix.txt`, and package layout.
4. Correlate absolute external paths with the system
   [toolchain inventory](toolchains.md#python).

The collector should not run arbitrary bundled Python code. If version metadata
is unavailable, report `unknown` rather than importing application modules.

## Packaging detection

Packaging tools leave different filesystem shapes:

| Packager | Strong markers | Useful fields |
|---|---|---|
| py2app | `Resources/__boot__.py`, `Resources/site.py`, `Resources/python-libraries.zip` | boot script, virtualenv prefix, package zip |
| PyInstaller | `base_library.zip`, `PYZ-*`, `*.pkg`, `lib-dynload/` | archive paths, extraction hints |
| Briefcase | `Resources/app_packages/`, `Resources/app/`, `pyvenv.cfg` | app module root, support package root |
| Nuitka | compiled extension-heavy layout, Nuitka metadata strings | Python version, compiler hints |
| Custom | launcher script references Python but no known packager layout | interpreter path, argv template |

Packager identification is evidence, not a security verdict. A custom launcher
may be perfectly legitimate; it simply needs different troubleshooting advice
than a self-contained PyInstaller bundle.

## Dependencies and native extensions

Python dependency inspection should prefer package metadata over filename
guessing:

- Read `*.dist-info/METADATA` and `*.egg-info/PKG-INFO` for package names and
  versions.
- Count top-level packages only when metadata is absent.
- Record native extensions (`.so`) separately from pure-Python packages.
- Run `otool -L` on native extensions only when deeper native dependency
  inspection is requested or the file count is small enough to stay cheap.

Native extensions are the Python analogue of Electron native modules: they are
where Python apps commonly cross into Swift, Objective-C, C, C++, Rust, GPU,
audio, screen-capture, database, and networking libraries. Spectra should
surface framework hints and risk hints in the same spirit as
[native module sub-detection](../detection/native-modules.md).

## Security and reliability checks

Python apps deserve a few Python-specific checks beyond ordinary codesign and
entitlements:

| Check | Why it matters |
|---|---|
| External interpreter path | App behavior can change when Homebrew, pyenv, uv, mise, or asdf changes |
| User-site enabled | Imports may come from mutable user directories |
| Writable module roots | Code loaded by the app can be modified by the current user |
| Unsigned native extensions | Hardened runtime may fail at load time, or the app may rely on disabled library validation |
| Quarantine on extracted payloads | First launch can fail or trigger confusing Gatekeeper prompts |
| `DYLD_*` / `PYTHONPATH` launcher usage | Environment-sensitive imports are hard to reproduce remotely |

These checks should become recommendation inputs rather than hard failures.
The practical output is a short explanation: "this app bundles Python 3.12 and
all packages" versus "this app launches `/opt/homebrew/bin/python3` and imports
from a writable support directory."

## JSON shape

Python app inspection is exposed as a nested field rather than overloading
`Dependencies`:

```go
type PythonApp struct {
    Packaging        string
    RuntimeSource    string
    RuntimePath      string
    Version          string
    ModuleRoots      []string
    Packages         []PythonPackage
    NativeExtensions []PythonNativeExtension
    RiskHints        []string
}
```

The existing top-level `UI`, `Runtime`, `Language`, and `Confidence` fields
continue to describe the main app verdict. Python details explain how that
verdict is implemented or launched.
